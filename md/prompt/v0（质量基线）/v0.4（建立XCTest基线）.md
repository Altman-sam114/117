# v0.4（建立 XCTest 基线）

## 版本号

v0.4

## 版本分配依据

人工未指定版本号。当前正式记录已到 `v0.3`，重点是 main 直推、GitHub Actions 云端重验证和 Agent C 结果包验收。本轮下一步优化目标是把 CI 里的 `testOutcome: skipped` 升级为真实 XCTest 结果，属于 `v0.x` 项目初始化与质量基线阶段的小版本递增，因此分配为 `v0.4`。

## 当前进展总结

- App 主功能已具备：本地 JSON 日记保存、列表、搜索、分类筛选、编辑器、Markdown 预览、`###` 小节、分享和统计看板。
- 核心架构边界已明确：`JournalStore` 是唯一集合修改和持久化入口，`JournalEntry` 是模型与兼容解码边界，`MarkdownBlockParser` 和 `JournalStatistics` 分别承担解析与统计规则。
- 协作制度已升级到 `v0.3`：Agent B 在 `main` 上实现并 push `origin/main`，GitHub Actions 生成未加密结果包，Agent C 下载结果包核对 manifest/JUnit/log。
- 当前测试基线仍偏弱：本地只有 `git diff --check`、Xcode project plist、Swift parse、workflow YAML 等轻量检查；CI 能做 generic iOS Debug build，但没有 XCTest target，manifest 中 `testOutcome` 仍为 `skipped`。
- 当前仓库实际已经配置 `origin/main`，但 `update_log.md` 的“当前远端状态”仍写着未配置 `origin`，下一轮需要同步修正文档状态，避免 Agent C 误判云端流程阻塞。

## 目标

1. 新增 `MDJournalTests` XCTest target，并纳入 `MDJournal.xcodeproj`。
2. 为纯 Swift 核心规则补第一批单元测试：
   - `JournalEntry`：旧 JSON 缺失 `updatedAt`、`category`、`mood` 时可兼容解码；空标题使用日期展示；`starterEntry` 包含默认 `###` 小节。
   - `JournalSection.extract`：只识别 `### ` 小节、空标题回退为“未命名小节”、正文摘要清理待办和 Markdown 标记。
   - `MarkdownBlockParser`：标题、段落、引用、无序列表、待办、代码块、分割线和未闭合代码块的解析结果稳定；`groupedByLevelThree` 能保留开篇块并按三级标题分组。
   - `JournalStatistics`：固定 calendar 和 now 时，总篇数、词数、平均值、连续天数、最长连续、最近 7 天趋势、分类/心情分布和小节覆盖率可预测。
   - `MarkdownSnippet`：每个 snippet 的 title、systemImage 和 markdown 不为空，关键 snippet 输出保持当前约定。
3. 更新 `.github/workflows/ci-results.yml`，在 generic build 外运行 XCTest，并让结果包记录真实测试 outcome。
4. 更新 `md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`README.md` 和 `update_log.md` 中受影响的测试基线、CI 结果包和当前远端状态。
5. 保持应用行为不变，不改变本地 JSON 格式，不引入第三方依赖。

## 非目标

- 不新增 App 功能。
- 不改 UI 视觉风格或交互结构。
- 不迁移到 SwiftData、CloudKit、服务器、账号或云同步。
- 不修改最低 iOS 版本 16.0。
- 不把 `JournalStore` 做大范围重构；如需要测试持久化，优先留作后续版本，避免本轮扩大到文件系统行为改造。
- 不新增 UI test 或模拟器交互测试，除非人工另行要求。
- 不创建 PR、不使用候选分支、不切换到 `develop`、`smalldata_test` 或 `codeb/...`。

## 架构依据

- `JournalEntry` 是兼容解码和派生展示属性的边界，测试应保护旧数据可读和 `###` 小节语义。
- `MarkdownBlockParser` 是轻量 Markdown 规则中心，测试应锁定当前支持的语法，不追求完整 CommonMark。
- `JournalStatistics` 是统计口径中心，视图只消费结果，因此统计测试应覆盖固定日期样例。
- SwiftUI 视图当前主要通过 binding 和 closure 触发状态变化，本轮不把视图行为放入 XCTest 第一批范围。
- `JournalStore` 当前初始化时直接定位 Documents 或临时目录并执行 `load()`，本轮不强行测试它的文件持久化，以免先重构存储 URL 注入；后续可单独设计 `JournalStore` 可测试性优化。

## 实现步骤

1. 开始前执行并记录：
   - `git fetch origin`
   - `git switch main`
   - `git pull --ff-only origin main`
   - `git status --short --branch`
2. 确认当前分支是 `main`，目标远端是 `origin/main`。如果缺少远端或无法同步，停止实现并记录阻塞。
3. 在 Xcode project 中新增 `MDJournalTests` unit test target：
   - product type 使用 `com.apple.product-type.bundle.unit-test`。
   - 依赖 `MDJournal` app target。
   - Debug/Release 均保持 iOS 16.0。
   - 不引入第三方依赖。
4. 新增测试文件，使用 `XCTest` 和 `@testable import MDJournal`。
5. 测试数据必须固定 UUID、日期、calendar 和 now，避免受当前日期、时区或运行顺序影响。
6. 优先覆盖纯逻辑：
   - JSON 解码用本地字符串和 `JSONDecoder.dateDecodingStrategy = .iso8601`。
   - 统计测试使用 `Calendar(identifier: .gregorian)`，设置稳定 timezone。
   - Markdown 测试直接比较 `[MarkdownBlock]` 和 `MarkdownSectionGroup`。
7. 更新 CI：
   - 保留 `git diff --check`、`plutil`、Swift parse 和 generic iOS Debug build。
   - 新增 `xcodebuild test`，建议使用 `-destination generic/platform=iOS Simulator` 或 GitHub macOS runner 可用的稳定模拟器 destination；如果 generic simulator 不支持测试，则用 `xcrun simctl list devices available` 选择固定可用 iPhone simulator。
   - 将测试日志写入 `ci-results/xctest.log`。
   - 将 XCTest `.xcresult` 写入结果包，例如 `ci-results/MDJournalTests.xcresult`。
   - manifest 新增或更新 `testOutcome`，成功为 `success`，失败为 `failure`，不再默认 `skipped`。
   - JUnit 至少反映 static checks、build 和 XCTest 三类结果；如没有成熟转换器，可先用脚本生成汇总 JUnit，但必须让失败数跟 outcome 一致。
   - artifact 名称建议递增到 `mdjournal-ci-v0.4-main-<short_sha>-run<run_id>-attempt<run_attempt>`。
8. 更新文档：
   - `md/test/test.md`：加入 XCTest target、测试命令、CI test outcome、结果包新增日志/xcresult、当前基线。
   - `md/flow/flow.md`：更新测试层关系和未来扩展点，说明核心规则已有 XCTest 基线。
   - `md/flow/flowchart.md`：更新 CI 结果包验收图和 Agent 云端迭代图里的测试节点。
   - `README.md`：补充本地轻量检查后可运行 XCTest 的命令，并说明云端会运行测试。
   - `update_log.md`：新增 `v0.4 / 建立 XCTest 基线` 记录；同步修正当前远端状态为已配置 `origin/main`，但不要写入包含 token 的远端 URL。

## 本地轻量检查要求

Agent B 完成后至少运行并记录结果：

```sh
git diff --check
plutil -lint MDJournal.xcodeproj/project.pbxproj
xcrun swiftc -parse -parse-as-library $(rg --files -g '*.swift' MDJournal)
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
```

因为本轮新增 XCTest target，还需要本机尝试运行测试命令。若当前本机模拟器或 CoreSimulator 不可用，必须记录关键错误，并说明最终以 GitHub Actions 结果包为准：

```sh
/Applications/Xcode.app/Contents/Developer/usr/bin/xcodebuild \
  -project MDJournal.xcodeproj \
  -scheme MDJournal \
  -configuration Debug \
  -destination 'platform=iOS Simulator,name=iPhone 16' \
  -derivedDataPath /private/tmp/mdjournal-derived-data \
  CODE_SIGNING_ALLOWED=NO \
  test
```

如果本机没有 `iPhone 16`，先用 `xcrun simctl list devices available` 查看可用设备，再选择一个可用 iPhone 模拟器。若 `xcrun simctl` 因 CoreSimulator 服务不可用失败，不得伪造本机测试通过。

## main 直推要求

- 只在 `main` 分支提交。
- 提交前用 `git status --short` 确认只包含本轮相关文件。
- 提交说明建议：`v0.4 建立 XCTest 基线`。
- 提交后执行：

```sh
git push origin main
```

- push 前确认目标远端是 `origin/main`，但不要在文档或最终回复里暴露任何远端 token。

## GitHub Actions workflow 和结果包要求

CI 成功或失败都必须上传未加密 artifact，至少包含：

- `ci-artifact-manifest.json`
- `ci-failure-summary.md`
- `static-checks.log`
- `xcodebuild.log`
- `xctest.log`
- `junit.xml`
- `MDJournal.xcresult`，如果 build 生成
- `MDJournalTests.xcresult`，如果 test 生成

manifest 至少包含：

- `version: v0.4`
- `branch`
- `commitSha`
- `shortSha`
- `runId`
- `runAttempt`
- `workflowName`
- `scheme`
- `destination`
- `staticChecksOutcome`
- `buildOutcome`
- `testOutcome`
- build/test 日志路径和 result bundle 路径

## Agent C 验收要求

Agent C 只能验收最新 `origin/main` commit 对应的 run 和 artifact，必须：

1. 查看 `origin/main` 最新 commit SHA。
2. 查看 `MD Journal CI Results` 最新 run id 和 run attempt。
3. 下载 artifact 到 `/private/tmp/mdjournal-c-review-<run_id>/`。
4. 核对 `ci-artifact-manifest.json` 中 `branch=main`，且 `commitSha`、`runId`、`runAttempt` 与最新 run 完全一致。
5. 核对 `junit.xml` 中 XCTest 不是 skipped，失败数与 manifest outcome 一致。
6. 阅读 `static-checks.log`、`xcodebuild.log`、`xctest.log` 和 `ci-failure-summary.md` 的关键结果。
7. 确认 `update_log.md` 已记录 v0.4 的验证结果、run id、run attempt、artifact 名称和遗留事项。

## 验收标准

- `MDJournalTests` target 存在并能被 CI 运行。
- 第一批测试覆盖 `JournalEntry`、`JournalSection`、`MarkdownBlockParser`、`JournalStatistics` 和 `MarkdownSnippet`。
- GitHub Actions manifest 的 `testOutcome` 不再固定为 `skipped`。
- CI artifact 包含测试日志或测试结果摘要。
- 文档中的测试基线、CI 结果包内容和当前远端状态与实际一致。
- App 主功能、JSON 格式、`###` 语义、分类/心情 rawValue、iOS 16.0 deployment target 均未被破坏。

## 风险和禁止项

- 禁止为了让测试通过而弱化或删除现有功能。
- 禁止无依据改动本地 JSON 字段、日期编码策略、分类和心情 rawValue。
- 禁止把测试写成只检查“不崩溃”的空测试；每个测试必须有明确断言。
- 禁止依赖当前日期、系统时区或测试顺序。
- 禁止在 CI 失败时仍把 manifest 写成 success。
- 禁止把旧 run、旧 artifact 或本地文件冒充本轮云端结果包。
- 禁止在最终回复或文档中暴露远端 URL 内的访问 token。
