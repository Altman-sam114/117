# v0.61 / Markdown 解析逐行迭代

## 版本号

`v0.61`

## 版本分配依据

当前最新正式版本是 `v0.60`。本轮继续沿 `v0.x` 小版本递增，目标贴合总目标中的“优美写入、高效的性能”，在 v0.60 已锁定代码块、空白行和 CR/CRLF 解析行为后，继续处理 `MarkdownBlockParser` 长文实时预览路径中的整篇分行数组分配。

## 背景

`MarkdownBlockParser.parse(_:)` 仍通过 `components(separatedBy: .newlines)` 一次性生成 `[String]`，长正文实时预览时会先为全篇正文创建行数组，即使很多行只是空白分隔、marker 行或代码块围栏。更合适的小步优化是使用与 Foundation newline 查找一致的逐行迭代，在保持现有分行语义的前提下，避免整篇行数组预分配，并让解析器只在需要持久进入 block 缓冲时把 `Substring` 转成 `String`。

## 目标

- 在 `MarkdownBlockParser` 内新增私有逐行组件迭代 helper，使用 `rangeOfCharacter(from: .newlines)` 查找下一段 newline 分隔符。
- `parse(_:)` 改为消费逐行 `Substring`，避免 `components(separatedBy: .newlines)` 生成整篇 `[String]`。
- 保持空文档、尾随换行、连续空行、CR-only、CRLF、代码块空行、代码块内 Markdown-like 行、有序列表和 `###` 小节分组行为不变。
- 补充 `MarkdownBlockParserTests` characterization，覆盖尾随换行、代码块尾随换行和 CRLF fenced code 行为。
- 更新 `.github/workflows/ci-results.yml` 的 `VERSION` 为 `v0.61`。
- 同步 README、测试规范、核心流程、流程图和更新日志。

## 非目标

- 不改 Markdown 支持范围，不引入 CommonMark 或第三方解析器。
- 不改变 heading、引用、无序列表、有序列表、待办、代码块、分割线或 `###` 小节分组规则。
- 不改变代码块内容 join 仍使用 `\n` 的当前行为。
- 不改变 `MarkdownPreviewView` 渲染、正文编辑、写作工具栏、JSON 持久化、统计口径、Xcode target 或 Mac Catalyst 构建方式。
- 不运行本机 app、模拟器、本机构建或 XCTest；最终以 GitHub Actions artifact 验收。

## 架构依据

- `MarkdownBlockParser` 是轻量 Markdown 分块与 `###` 小节分组边界，分行优化应留在该模块内部。
- `MarkdownPreviewView` 只消费解析结果，不应知道 parser 如何分行。
- `MarkdownBlockParserTests` 是解析行为的契约，本轮必须先用 characterization 锁定边界，再替换分行入口。

## 实现步骤

1. 在 `MarkdownBlockParser` 中新增私有 helper，例如 `forEachLineComponent(in:_:)`，从 `markdown.startIndex` 开始用 `rangeOfCharacter(from: .newlines, range:)` 找 separator，逐个回调 `Substring`。
2. 确保 helper 在空字符串和尾随 newline 时仍会回调空行组件，匹配 `components(separatedBy: .newlines)` 对 parser 的有效语义。
3. 将 `parse(_:)` 的 `for rawLine in lines` 改为调用 helper；`paragraphLines` 和 `codeLines` 仍保存 `[String]`，只在需要进入缓冲时执行 `String(rawLine)`。
4. 将 `isHorizontalWhitespaceOnly` 和 `leadingWhitespaceTrimmed()` 扩展改为适配 `Substring` 或 `StringProtocol`，避免 marker 检查前把整行转成 `String`。
5. 增加尾随换行、代码块尾随换行和 CRLF fenced code tests，确认本轮不改变当前语义。
6. 同步 workflow 版本号和文档。
