# v0.56 / Markdown 行缩进扫描范围收窄

## 版本分配依据

当前最新正式记录为 `v0.55`。本轮继续服务 Agent X 总目标中的“高效性能”和“优美写入”，在 v0.55 已经复用行首 UTF-16 offset 的基础上，继续缩小正文 Tab / Shift-Tab 行缩进的扫描范围，因此递增为 `v0.56`。

## 背景

`MarkdownLineIndentation.apply` 是正文编辑中的高频写作辅助路径。v0.55 已经把每个 operation 的行首 UTF-16 location 改为复用行首表，避免每行从正文开头重复计算 `NSRange`。但当前 `lineStarts(in:)` 仍会为整篇正文收集所有行首；当用户只在长文靠前位置按 Tab / Shift-Tab 时，仍会扫描不相关的后续正文。

## 本轮目标

- 将 `MarkdownLineIndentation` 的行首扫描范围限制到选区有效结束行。
- 保持同一次扫描收集行首 `String.Index` 和 UTF-16 offset，继续复用 v0.55 的 operation location 设计。
- 保持既有行为：Tab 插入两个空格；Shift-Tab 删除一个 tab、两个行首空格或一个行首空格；非空选区用 `location + length - 1` 判断有效结束；选区刚好结束在下一行行首时不处理下一行；emoji/UTF-16 光标和选区边界保持正确。
- 增加最小 XCTest 覆盖长正文靠前光标/选区、尾随空行光标和 UTF-16/emoji 边界，证明扫描范围收窄不改变语义。

## 非目标

- 不改变缩进单位、快捷键、菜单命令、工具栏或 UI。
- 不改变 CR/CRLF 分行语义。
- 不重写多次 `replaceSubrange` 为一次性构造新字符串。
- 不改变 Markdown 片段插入、回车续写、预览解析、统计、列表或 JSON 持久化。
- 不引入第三方依赖。

## 架构依据

- `MarkdownLineIndentation` 是纯规则工具，仍由 `MarkdownBodyTextView`、`EntryEditorView.applyIndentation(_:)`、Mac Catalyst 写作菜单和写作工具栏复用。
- `JournalStore` 仍是唯一持久化入口，本轮不触碰数据模型或保存逻辑。
- 行缩进测试继续放在 `MDJournalTests/MarkdownLineIndentationTests.swift`。

## 实现步骤

1. 在 `apply` 中先计算 clamped `NSRange` 的 effective end offset：折叠选区用 `location`，非折叠选区用 `max(location, location + length - 1)`。
2. 将行首扫描 helper 改为接收 effective end offset，并在扫描到包含该 offset 的行后停止，不再遍历后续正文。
3. 保证 helper 至少返回正文起始行；如果 effective end 位于 LF 后的尾随空行，也必须返回该尾随空行行首。
4. `selectedLineStarts(for:from:)` 继续从已收集行首中用二分定位首尾行；因为扫描已覆盖 effective end 所在行，不能越界。
5. 不改 `adjustedBoundary`、缩进/反缩进规则或替换顺序。
6. 增加测试覆盖长正文靠前光标缩进、尾随换行末尾光标缩进和非零起点跨 emoji 行选区。
