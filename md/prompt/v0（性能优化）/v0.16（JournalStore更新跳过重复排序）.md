# v0.16（JournalStore 更新跳过重复排序）

## 版本号

v0.16

## 版本分配依据

当前最新正式版本为 `v0.15`，已完成 Markdown 片段按光标/选区插入。人工总目标要求继续推进“Mac 版、优美写入、高效性能”。当前 `JournalStore.update(_:)` 在每次标题、正文、分类、心情或日期 binding 写回时都会重新排序 `entries`，但项目排序规则只依赖 `createdAt`。正文输入是最高频写作路径，本轮沿 `v0.x` 小版本递增为 `v0.16`，优先移除正文输入中的无效排序成本。

## 背景

`JournalStore` 已经在 v0.7 引入短延迟合并写盘，JSON 写入不会每个字符都落盘。但正文通过 `MarkdownBodyTextView` 写入 `EntryEditorView` 的 `entry.body` binding 后，仍会进入 `ContentView.selectedEntryBinding` 的 setter，并调用 `JournalStore.update(_:)`。当前 `update(_:)` 每次都会替换数组元素、更新 `updatedAt`、执行 `sortEntries()` 并安排保存。

由于当前唯一排序规则是 `createdAt` 倒序，正文、标题、分类、心情和 `updatedAt` 变化不会改变列表顺序。每次输入都排序会增加不必要的主线程工作，也会让后续列表派生、统计派生更容易在频繁发布下放大成本。本轮不改变“内存状态即时更新”和 `JournalStore` 作为唯一修改入口的边界，只让排序按需发生。

## 目标

- 修改 `JournalStore.update(_:)`，仅当 `createdAt` 改变时才重新排序。
- 正文、标题、分类、心情等不影响排序的更新仍必须即时写入 `entries` 并安排 debounced save。
- `createdAt` 被 DatePicker 修改时仍必须保持 `entries` 按 `createdAt` 倒序。
- 保持 `createEntry()`、`delete(_:)`、`load()` 的排序和保存语义不变。
- 补充 `JournalStoreTests`，证明正文更新不会改变既有排序、日期更新会重新排序。
- GitHub Actions 结果包版本更新为 `v0.16`。

## 非目标

- 不实现“本地正文草稿 + 延迟提交”方案；该方案会改变 `JournalStore.entries` 共享状态即时更新时序，需单独设计。
- 不改变 JSON schema、本地文件名、编码格式或保存路径。
- 不改变 `JournalEntry` 的排序规则，仍按 `createdAt` 倒序。
- 不绕过 `JournalStore` 直接批量改写日记集合。
- 不引入第三方依赖、数据库、持久化索引、账号、服务器或云同步。
- 不新增 native macOS target；Mac 路径仍是 Mac Catalyst。

## 架构依据

- `JournalStore` 是唯一的日记集合修改和本地持久化入口。
- `EntryEditorView` 和 `MarkdownBodyTextView` 只负责编辑输入与选区同步，不应承担集合排序策略。
- `ContentView.selectedEntryBinding` 继续把编辑器变更交给 `JournalStore.update(_:)`。
- `JournalEntry.createdAt` 是当前唯一影响列表排序的字段。
- `JournalStoreTests` 是保存、flush 和更新排序行为的回归测试边界。

## 实现步骤

1. 在 `JournalStore.update(_:)` 中读取当前 stored entry，比较旧 `createdAt` 与新 `entry.createdAt`。
2. 继续设置 `updatedAt = Date()` 并替换 `entries[index]`，确保内存状态即时更新。
3. 仅当 `createdAt` 改变时调用 `sortEntries()`。
4. 保持 `scheduleSave()` 调用不变，确保所有更新仍进入 debounced save。
5. 扩展 `JournalStoreTests`：
   - 构造两个不同 `createdAt` 的 entry，确认修改较旧 entry 的 `body` 后顺序不改变。
   - 修改较旧 entry 的 `createdAt` 到更新日期，确认 `update(_:)` 后该 entry 移到前面。
6. 更新 CI `VERSION` 为 `v0.16`。
7. 同步 README、测试规范、核心流程、流程图和更新日志，说明本轮优化的是 `JournalStore.update(_:)` 的按需排序，不是写盘节流重做。

## 风险和禁止项

- 禁止改变 `JournalEntry` JSON schema 或本地持久化文件格式。
- 禁止让正文更新绕过 `JournalStore.update(_:)`。
- 禁止为了减少发布频率而牺牲 `JournalStore.entries` 的即时内存更新。
- 禁止改变 `createdAt` 倒序排序规则。
- 禁止用旧 run 或旧 artifact 冒充本轮 v0.16 云端验收。
