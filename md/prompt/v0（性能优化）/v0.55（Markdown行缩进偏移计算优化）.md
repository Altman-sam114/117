# v0.55 / Markdown 行缩进偏移计算优化

## 版本分配依据

当前最新正式记录为 `v0.54`。本轮继续服务 Agent X 总目标中的“优美写入”和“高效性能”，选择正文 Tab / Shift-Tab 行缩进规则里的 UTF-16 偏移计算作为小范围性能优化，因此递增为 `v0.55`。

## 背景

`MarkdownLineIndentation.apply` 是正文编辑中的高频写作辅助路径。当前实现会为选区起止位置做 UTF-16 offset 到 `String.Index` 的扫描，并在每个行首 operation 中通过 `NSRange(body.startIndex..<lineStart, in: body).length` 重新计算行首 UTF-16 location。长正文多行选区缩进或反缩进时，这会产生重复扫描成本。

## 本轮目标

- 让 `MarkdownLineIndentation` 在一次行首扫描中记录每个行首的 `String.Index` 与 UTF-16 location。
- 选区命中的首尾行、operation location 和选区更新复用同一份行首位置信息。
- 保持现有语义：Tab 插入两个空格；Shift-Tab 删除一个 tab、两个行首空格或单个行首空格；多行选区只处理被选中的行；选区刚好结束在下一行行首时不处理下一行；UTF-16/emoji 选区边界保持正确。
- 增加最小 XCTest 覆盖长多行选区与 emoji/UTF-16 混合边界，证明优化不改变行为。

## 非目标

- 不改变缩进单位、快捷键、菜单命令或 UI。
- 不改变 Markdown 片段插入、回车续写、预览解析、统计、列表或 JSON 持久化。
- 不重写正文 bridge 或 Mac Catalyst 构建/运行脚本。
- 不引入第三方依赖。

## 架构依据

- `MarkdownLineIndentation` 是纯规则工具，仍由 `MarkdownBodyTextView`、`EntryEditorView.applyIndentation(_:)`、Mac Catalyst 写作菜单和写作工具栏复用。
- `JournalStore` 仍是唯一持久化入口，本轮不触碰数据模型或保存逻辑。

## 实现步骤

1. 在 `MarkdownLineIndentation` 内新增轻量行首结构，包含 `index: String.Index` 与 `location: Int`。
2. 用单次从 `text.startIndex` 到 `text.endIndex` 的扫描构造行首表，初始包含正文起点，遇到 LF 后追加下一行起点和当前 UTF-16 offset。
3. 用行首表根据 clamped `NSRange` 找到选区首尾行；非空选区仍使用 `location + length - 1` 作为有效结束 offset，保持“结束在下一行行首不缩进下一行”的语义。
4. `operation` 接收行首结构，直接使用其 `location`，不再为每行构造 `NSRange(body.startIndex..<lineStart, in: body)`。
5. 删除已无用的 offset 转 index、向前找行首、向后找下一行 helper。
6. 增加或调整测试，覆盖长多行选区、emoji UTF-16 offset、末尾行和下一行行首边界。
