# v0.7（编辑写入节流）

## 版本号

v0.7

## 版本分配依据

人工总目标要求“优美的写入，高效的性能”。`v0.6` 已建立 Mac Catalyst 可构建基线并通过 artifact 验收，但当前编辑链路仍是 `TextEditor` 每次输入 -> `Binding<JournalEntry>` setter -> `JournalStore.update(_:)` -> JSON 立即写盘。长文本和 Mac 键盘高速输入会放大同步编码和原子写入成本，因此本轮继续在 `v0.x` 阶段做窄范围性能优化。

## 背景

当前 `JournalStore.update(_:)` 会在每次编辑标题、日期、分类、心情或正文后立即保存整个 `[JournalEntry]` 到 `Documents/md-journal-entries.json`。这保证数据安全，但对正文输入不够高效。需要保持 SwiftUI 状态实时更新，同时把磁盘写入合并为短延迟保存，并在应用离开前强制 flush。

## 本轮目标

- `JournalStore.update(_:)` 仍立即更新内存中的 `entries`，保持 UI 实时响应。
- 编辑更新默认节流保存，短时间连续输入只合并为一次 JSON 写盘。
- 创建、删除和应用进入 inactive/background 时必须立即 flush，降低数据丢失风险。
- 保存失败仍写入 `errorMessage`，不静默吞错。
- 新增 XCTest 覆盖节流保存和 flush 行为。
- 不改变 JSON 文件名、字段、编码策略、兼容解码或排序规则。
- 同步更新 README、测试规范、核心流程、流程图和更新日志中受影响部分。

## 非目标

- 不引入数据库、Core Data、服务器、账号或云同步。
- 不改变 `JournalEntry` 数据格式。
- 不把持久化逻辑移动到视图层。
- 不改 Markdown 解析、预览语法或统计口径。
- 不做大范围并发重构；仍保持 `JournalStore` 为 `@MainActor ObservableObject`。

## 架构依据

- `JournalStore` 是唯一日记集合修改与本地持久化入口，节流逻辑必须留在 store 内。
- `ContentView` 只负责在 scene phase 变化时请求 store flush，不直接写文件。
- `EntryEditorView` 仍只通过 binding 编辑 `JournalEntry`。
- 测试可通过注入临时 storage URL 和较短 debounce 间隔验证行为，但不得改变生产默认存储路径。

## 实现步骤

1. 扩展 `JournalStore` 初始化能力，允许测试注入 `storageURL` 和较短 debounce 间隔；生产默认仍使用 Documents 下 `md-journal-entries.json`。
2. 将编辑更新保存改为节流：
   - `update(_:)` 更新内存、更新时间、排序后调用 `scheduleSave()`。
   - `createEntry()` 和 `delete(_:)` 使用立即保存。
   - `flushPendingSave()` 取消待执行任务并立即写入当前 entries。
3. 使用 `Task.sleep` 实现短延迟保存，任务结束后回到 `@MainActor` 执行保存。
4. 在 `ContentView` 监听 `scenePhase`，当 scene phase 不是 `.active` 时调用 `store.flushPendingSave()`。
5. 新增 `JournalStoreTests`：
   - 创建日记应立即写入磁盘。
   - `update(_:)` 应立即更新内存但延迟写入磁盘。
   - `flushPendingSave()` 应立即写入待保存变更。
6. 更新 Xcode project，把 `JournalStoreTests.swift` 加入 `MDJournalTests` target。
7. 将 CI 结果包 manifest 和 artifact 命名版本更新为 `v0.7`，方便 Agent C 核对本轮云端结果。
8. 同步文档和 update log。

## Agent C 云端验收要求

Agent C 必须下载最新 run 的 artifact，并核对：

- manifest 的 `branch`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致。
- `staticChecksOutcome`、`buildOutcome`、`macCatalystBuildOutcome`、`testOutcome` 全部为 `success`。
- `junit.xml` 无 failure，XCTest 不 skipped，测试数量应覆盖新增 `JournalStoreTests`。
- `xcodebuild.log`、`maccatalyst-build.log`、`xctest.log` 结尾无失败。

## 风险和禁止项

- 禁止因为节流导致创建或删除丢失。
- 禁止应用进入后台时仍留有未写盘变更。
- 禁止把写盘延迟做得过长，生产默认应是短延迟。
- 禁止改变旧 JSON 解码兼容性。
- 禁止在测试里写入用户 Documents 目录。
