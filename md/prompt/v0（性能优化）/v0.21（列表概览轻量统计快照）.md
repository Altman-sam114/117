# v0.21（列表概览轻量统计快照）

## 版本号

`v0.21`

## 版本分配依据

当前最新正式版本为 `v0.20`。本轮继续沿 `v0.x` 性能优化推进，目标聚焦编辑正文时列表首页的同步重算成本。

## 背景

`EntryListView.body` 当前会直接构造完整 `JournalStatistics(entries:)`，但列表概览卡只展示总篇数、连续天数、总词数和一段 insight。完整统计看板还会计算最长连续天数、本周统计、最近 7 天趋势、心情分布等数据。正文编辑会即时更新 `JournalStore.entries`，从而触发列表刷新；列表首页不应每次都付出完整统计看板的计算成本。

## 目标

- 新增纯规则 `JournalListOverviewSnapshot`，只计算列表概览需要的字段。
- `EntryListView` 概览卡改用轻量快照，不再在列表 body 中构造完整 `JournalStatistics`。
- 保持概览卡展示语义与原 `JournalStatistics` 一致，包括空状态、连续记录、低小节覆盖率、主导分类和本周兜底文案。
- 新增 `JournalListOverviewSnapshotTests`，对照 `JournalStatistics` 验证关键字段和 insight 文案一致。
- 将 CI artifact 版本更新为 `v0.21`，并同步 README、测试规范、核心流程、流程图和更新日志。

## 非目标

- 不改变统计看板 `StatisticsDashboardView`，完整看板继续使用 `JournalStatistics`。
- 不改变 `JournalStatistics` 对外字段或统计口径。
- 不改变 `JournalEntryBodySummary` 的词数、摘要或 `###` 小节识别语义。
- 不改变 `JournalStore` 保存、排序、节流或 JSON 持久化。
- 不引入缓存失效机制、第三方依赖或 native macOS target。

## 架构依据

- `EntryListView` 负责列表和概览展示，但不应承载完整统计看板计算。
- `JournalListOverviewSnapshot` 属于 Utilities 纯派生规则，只消费 `[JournalEntry]`、`Calendar` 和 `now`。
- `JournalStatistics` 继续负责完整统计看板，避免破坏既有测试和视图边界。

## 实现步骤

1. 新增 `JournalListOverviewSnapshot`：
   - 字段包含 `totalEntries`、`totalWords`、`entriesWithSections`、`sectionCoverage`、`recentStreak`、`entriesThisWeek`、`wordsThisWeek`、`dominantCategory` 和 `insightText`。
   - 单轮遍历 entries，避免排序、心情分布、7 天趋势和最长连续天数计算。
   - insight 文案与 `JournalStatistics.insightText` 保持一致。
2. 修改 `EntryListView`：
   - `body` 中构造 `JournalListOverviewSnapshot(entries:)`。
   - `overviewCard` 参数从 `JournalStatistics` 改为 `JournalListOverviewSnapshot`。
3. 新增 `JournalListOverviewSnapshotTests`：
   - 空列表字段和 insight。
   - 固定 calendar/now 下与 `JournalStatistics` 的概览字段一致。
   - 连续 3 天优先返回连续记录 insight。
   - 低小节覆盖率 insight。
   - 主导分类 insight 与 entryCount / wordCount tie-break 规则。
4. 更新 Xcode project，把新 utility 加入 app target，把新 test 加入 `MDJournalTests`。
5. 更新 workflow `VERSION: v0.21`。
6. 同步 README、测试规范、核心流程、流程图和更新日志。

## 本地轻量检查要求

人工已明确要求本轮不要在本机运行构建、测试或 app。因此 Agent B 不执行本机 `xcodebuild build`、`xcodebuild test`、模拟器或 app 启动；最终验收只以 GitHub Actions 回传 artifact 为准。

可记录只读命令、git 状态、diff 复核和提交/推送命令；不得用本地输出冒充云端验收。

## main 直推要求

- 默认分支：`main`。
- 目标远端：`origin/main`。
- 开始前执行 `git fetch origin`、`git pull --ff-only origin main`。
- 不创建候选分支，不创建 PR。

## GitHub Actions workflow 和结果包要求

- workflow 名称：`MD Journal CI Results`。
- artifact 名称应包含 `v0.21`、`main`、短 SHA、run id 和 run attempt。
- 结果包至少包含 manifest、failure summary、主日志、JUnit 和 `.xcresult`。

## Agent C 验收要求

Agent C 必须下载最新 `origin/main` commit 对应的未加密 artifact，并核对：

- manifest 中 `version=v0.21`。
- `branch=main`。
- `commitSha` 等于最新 `origin/main`。
- `runId`、`runAttempt` 与实际 GitHub Actions run 一致。
- `staticChecksOutcome`、`buildOutcome`、`macCatalystBuildOutcome`、`testOutcome` 均为 `success`。
- `junit.xml` failures 为 0，skipped 为 0。
- `xcodebuild.log` 和 `maccatalyst-build.log` 包含 `** BUILD SUCCEEDED **`。
- `xctest.log` 包含 `** TEST SUCCEEDED **`，且能看到 `JournalListOverviewSnapshotTests` 执行。
