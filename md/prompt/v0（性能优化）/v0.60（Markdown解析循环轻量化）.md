# v0.60 / Markdown 解析循环轻量化

## 版本号

`v0.60`

## 版本分配依据

当前最新正式版本是 `v0.59`。本轮继续沿 `v0.x` 小版本递增，目标贴合总目标中的“优美写入、高效的性能”，优先处理长文实时预览中 `MarkdownBlockParser` 的低风险循环和缓冲分配成本。

## 背景

`MarkdownBlockParser.parse(_:)` 仍使用 `components(separatedBy: .newlines)`，这是长正文预览解析的剩余较大分配点；但直接替换整套分行策略会牵涉 CR / CRLF 既有语义，风险较高。当前更安全的小步优化是：在保持现有分行方式不变的前提下，减少每行不必要的扫描，并让 flush 后的临时数组保留容量，降低多段落、多列表、多代码块文档反复解析时的数组容量 churn。

## 目标

- 在 `MarkdownBlockParser.parse(_:)` 中先处理 fenced code 状态，让代码块内行不再计算普通空白分隔逻辑。
- 在非代码块中先判断水平空白行，空白行直接 flush，不再额外执行行首裁剪。
- 将 parser 内部临时数组 flush 后的 `removeAll()` 改为 `removeAll(keepingCapacity: true)`，复用段落、列表、待办、代码和小节分组缓冲容量。
- 补充 `MarkdownBlockParserTests` characterization：代码块内空行保留、代码块内 Markdown-like 行不参与列表/小节、CR-only 和 CRLF 当前分行行为。
- 更新 `.github/workflows/ci-results.yml` 的 `VERSION` 为 `v0.60`。
- 同步 README、测试规范、核心流程、流程图和更新日志。

## 非目标

- 不替换 `components(separatedBy: .newlines)`，不引入自定义分行器。
- 不改变 CR / CRLF / CR-only 既有解析语义。
- 不改变 heading、引用、无序列表、有序列表、待办、代码块、分割线或 `###` 小节分组规则。
- 不改变 `MarkdownPreviewView` 渲染策略、正文编辑、写作工具栏、JSON 持久化或统计口径。
- 不运行本机 app、模拟器、本机构建或 XCTest；最终以 GitHub Actions artifact 验收。

## 架构依据

- `MarkdownBlockParser` 负责轻量 Markdown 分块与 `###` 小节分组，是长文预览性能路径。
- `MarkdownPreviewView` 只消费 `MarkdownParseResult`，不应承载解析优化。
- `MarkdownBlockParserTests` 已覆盖解析规则，本轮继续在该文件补充现有语义 characterization，避免后续自定义分行器误改行为。

## 实现步骤

1. 调整 `MarkdownBlockParser.parse(_:)` 的 per-line 处理顺序：`isReadingCode` 分支优先，只执行 closing fence 所需的行首裁剪；非代码块再判断空白行；非空行才执行行首裁剪和 marker 识别。
2. 将 `paragraphLines`、`unorderedItems`、`orderedItems`、`checklistItems`、`codeLines` 和 `currentBlocks` 的 flush 清理改为 `removeAll(keepingCapacity: true)`。
3. 补充代码块内空行和 Markdown-like 行测试，确认代码块内容不触发普通空白 flush、列表解析或 `###` 小节分组。
4. 补充 CR-only 和 CRLF 当前行为锁定测试，明确本轮没有改变 `.newlines` 分行语义。
5. 同步 workflow 版本号和文档。

