# v0.50（Markdown 选区片段转换单次扫描）

## 版本号

`v0.50`

## 版本分配依据

- 人工指定目标版本为 `v0.50`，本轮沿当前 `v0.x` 性能优化阶段继续递增。
- 当前 `update_log.md` 和最新提示词已记录到 `v0.49`，本轮聚焦 `MarkdownSnippetInsertion` 的内部构造成本优化，不改变 Markdown 片段插入的外部语义，因此不开新大版本。

## 背景

`MarkdownSnippetInsertion` 负责正文工具栏、Mac Catalyst “插入 Markdown”菜单、写作工具栏和快捷键入口共用的片段插入规则。当前引用、无序列表、有序列表和待办对多行选区做转换时，会先执行：

```swift
selectedText.components(separatedBy: "\n")
```

再通过 `map` 生成每行的新字符串，最后 `joined(separator: "\n")` 合并。这个路径语义清晰，但对长选区会产生临时数组、每行中间字符串和最终 joined 中间过程。Mac Catalyst 桌面菜单、写作工具栏和正文工具栏都会触发同一规则，长文多行选区转换时可以进一步减少分配。

本轮要求用单次扫描或增量构造 replacement text 替代上述 `components + map + joined` 路径，同时保持现有 `MarkdownSnippetTests` 覆盖的所有契约，尤其是空白行跳过、尾随换行、有序列表编号、选区范围和 UTF-16/emoji 边界。

## 目标

- 优化 `MarkdownSnippetInsertion` 中引用、无序列表、待办和有序列表对非空多行选区的转换实现。
- 替代 `selectedText.components(separatedBy: "\n") + map + joined(separator: "\n")` 路径，改为单次扫描选区文本并增量构造 replacement text。
- 保持既有语义：
  - 空白行跳过，不添加 `> `、`- `、`- [ ] ` 或 `N. ` 前缀。
  - 尾随 `\n` 保留，且不额外生成空列表项、空引用项或空待办项。
  - 有序列表只给非空行递增编号，从 `1. ` 开始连续编号。
  - 替换后的 `selectedRange` 仍从 replacement 起点开始覆盖 replacement 全文。
  - `NSRange`、UTF-16、emoji 和无效 UTF-16 range clamp 行为不变。
  - blank line 判断必须与现有 `String.isBlankLine` 的 `trimmingCharacters(in: .whitespacesAndNewlines).isEmpty` 语义等价。
  - 不意外改变 `\r`、`\r\n` 或其它空白行行为；扫描分行边界仍应等价于原 `components(separatedBy: "\n")`，也就是只按 LF 切分，不把 CR 单独当成换行分隔符。
- 补充 `MarkdownSnippetTests`，锁定长选区转换的关键语义和 CR/emoji 边界，避免性能优化引入行为漂移。
- 同步 workflow 版本、README、测试规范、核心流程、流程图和更新日志中受影响的当前状态描述。

## 非目标

- 不改变空选区插入规则，也不改变 heading、bold、italic、code、divider 的 replacement 逻辑。
- 不改变 `clampedRange(for:in:)`、`clampedNSRange(_:in:)` 或 UTF-16 边界修正算法。
- 不改变 `MarkdownSnippet` 的菜单顺序、标题、图标、快捷键或 `markdown` 原始片段。
- 不改变 `EntryEditorView.insertSnippet(_:)`、Mac Catalyst command/focused value 路由、写作工具栏入口或正文 toolbar 入口。
- 不改变 `MarkdownLineContinuation`、`MarkdownLineIndentation`、Markdown 预览、统计、列表筛选、`JournalStore` 保存或本地 JSON 格式。
- 不引入第三方依赖，不修改 deployment target，不做无关重构。

## 架构依据

- `MarkdownSnippetInsertion` 是片段插入的纯规则层，输入 `MarkdownSnippet`、正文 `String` 和 UTF-16 `NSRange`，输出更新后的正文和选区。
- `EntryEditorView`、正文工具栏、Mac Catalyst “插入 Markdown”菜单和写作工具栏都复用这套规则；因此性能优化应集中在 `MarkdownSnippetInsertion`，不复制到 UI 层。
- `JournalStore` 是唯一持久化入口；本轮只改变编辑 replacement 的临时构造方式，不涉及日记集合修改或 JSON 解码/编码边界。
- Markdown 预览由 `MarkdownBlockParser` 和 `MarkdownPreviewView` 负责；本轮只影响插入/转换正文文本，不改变预览解析规则。

## 影响范围和关键文件

- `MDJournal/Utilities/MarkdownSnippetInsertion.swift`
  - 重写 `prefixedLineReplacement(prefix:selectedText:)`，用单次扫描/增量构造 replacement text 替代 `components + map + joined`。
  - 重写 `orderedListReplacement(selectedText:)`，在同一类扫描中只为非空行递增编号。
  - 可提取私有 helper 复用“按 LF 扫描每一行、判断 blank line、追加前缀/编号、保留分隔符”的逻辑，但 helper 必须保持局部、可读，不扩大抽象层级。
  - blank line 判断必须等价于旧 `String.isBlankLine`：对一行内容执行 `.whitespacesAndNewlines` trim 后为空即 blank。实现时可以直接扫描 unicode scalar 并用 `CharacterSet.whitespacesAndNewlines` 判断，避免构造 trimmed 字符串；不要用 `Character.isWhitespace` 替代，避免语义扩大或缩窄。

## 实现步骤

1. 开始前按项目规则读取入口文档、测试规范、本提示词和相关源码，并同步最新 `origin/main`。
2. 在 `MarkdownSnippetInsertion.swift` 中定位 `prefixedLineReplacement(prefix:selectedText:)` 和 `orderedListReplacement(selectedText:)`。
3. 设计一个只按 LF 扫描的 replacement 构造路径。建议形态：
   - 用 `var text = String()` 增量追加结果。
   - 维护当前行起点 `lineStart`，遍历 `selectedText.indices`，遇到 `"\n"` 时处理 `lineStart..<newlineIndex`，再追加 `"\n"`，并把下一位置作为新行起点。
   - 循环结束后只在 `lineStart < selectedText.endIndex` 时处理最后一行；这样尾随 LF 不会额外处理一个空行，也不会生成空列表项。
   - 行内容为空或 blank 时原样追加该行，不加前缀/编号。
   - 非 blank 行追加固定前缀，或对有序列表追加 `"\(nextNumber). "` 后递增编号。
4. blank line helper 必须语义等价于旧实现：
   - 空行返回 `true`。
   - 行内所有 unicode scalar 都属于 `CharacterSet.whitespacesAndNewlines` 时返回 `true`。
   - 包含任意非 `.whitespacesAndNewlines` scalar 时返回 `false`。
   - 不使用 `trimmingCharacters(in:)`、`components(separatedBy:)`、`split`、`map`、`joined` 来完成本轮目标路径。
5. 保持 `Replacement.selectedRange = NSRange(location: 0, length: text.utf16.count)`，确保选区覆盖 replacement 全文。
6. 保持 `replacement(for:)`、`emptySelectionReplacement(for:)`、`wrappingReplacement`、`codeBlockReplacement` 和 UTF-16 clamp helper 的外部行为不变。
7. 扩展 `MarkdownSnippetTests`：
   - 现有 `testSnippetInsertionSkipsBlankLinesWhenPrefixingSelectedLines`、`testSnippetInsertionPreservesTrailingNewlineWithoutExtraEmptyBullet`、`testSnippetInsertionPrefixesSelectedLinesWithOrderedNumbers`、`testSnippetInsertionSkipsBlankLinesWhenNumberingSelectedLines`、`testSnippetInsertionPreservesTrailingNewlineWithoutExtraOrderedItem`、`testSnippetInsertionLeavesBlankOnlySelectionsUnmarked` 必须继续通过。
   - 增加 CRLF 输入测试，例如 `第一行\r\n\r\n第二行\r\n`，确认 CRLF 被保留，CR-only blank line 不加前缀，尾随 LF 不生成额外项。
   - 增加只含 `\r` 或空格 + `\r` 的空白行测试，确认 blank 判断仍等价于 `.whitespacesAndNewlines`。
   - 增加有序列表在 CRLF/空白行之间只给非空行连续编号的测试。
   - 增加包含 emoji 的多行选区测试，确认 replacement 文本和 `selectedRange.length` 使用 UTF-16 计数且覆盖全文。
8. 更新 `.github/workflows/ci-results.yml` 的 `VERSION` 为 `v0.50`。
9. 同步 README、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md` 中受影响描述，避免文档落后于真实行为。
10. 只提交本轮相关文件，不扩大范围。

## 风险和禁止项

- 禁止改变空白行跳过、尾随 LF 保留、有序列表编号、replacement 选区覆盖全文或 UTF-16/emoji 边界行为。
- 禁止把分行规则从原来的 LF 切分改成按 `Character.isNewline`、`split(whereSeparator:)` 或 CR 切分；CR 必须继续作为行内容的一部分参与 blank 判断和输出。
- 禁止把 blank line 判断从 `.whitespacesAndNewlines` 语义改成 `.whitespaces`、`Character.isWhitespace` 或只识别普通空格/tab。
- 禁止在本轮目标路径中继续使用 `components(separatedBy: "\n")`、`map` 和 `joined(separator: "\n")` 构造多行选区 replacement。
- 禁止引入性能时间断言、随机测试或依赖具体机器速度的测试。
- 禁止改变空选区插入、包裹选区、代码块插入、divider 插入或 Mac Catalyst command 路由。
- 禁止绕过 `JournalStore` 或改动本地 JSON 持久化格式。
- 禁止引入第三方依赖、修改 deployment target 或扩大到 Markdown 预览/统计/列表重构。
- 禁止把旧 artifact、旧输出或本地文件冒充本轮云端结果包。
- 若云端验证或 artifact 下载权限受阻，必须明确记录阻塞，不能伪装 Agent C 已验收。
