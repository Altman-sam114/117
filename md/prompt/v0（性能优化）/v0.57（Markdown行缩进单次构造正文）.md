# v0.57 / Markdown 行缩进单次构造正文

## 版本号

`v0.57`

## 版本分配依据

当前最新正式版本是 `v0.56`。本轮继续沿 `v0.x` 性能优化阶段递增，目标是减少长文多行 Tab / Shift-Tab 时的正文字符串改写成本。

## 背景

`MarkdownLineIndentation` 已在 `v0.55` 复用行首 UTF-16 offset，并在 `v0.56` 把行首扫描范围收窄到选区有效结束行。当前剩余热点是 `apply(...)` 先生成多条 operation 后，再对 `updatedBody` 逐条反向 `replaceSubrange`。长多行选区会对同一个 `String` 触发多次变更。

## 目标

- 将多条缩进/反缩进 operation 的正文改写改为基于原始 `body` 和升序 operation 的单次构造。
- 保留当前 `updatedRange(_:applying:)` 的 UTF-16 选区调整规则。
- 补充测试覆盖长多行混合缩进/反缩进选区，确保正文和 selectedRange 与既有语义一致。
- 更新 `.github/workflows/ci-results.yml` 的 `VERSION` 为 `v0.57`。
- 同步 README、测试规范、核心流程、流程图和更新日志。

## 非目标

- 不改变缩进单位，仍为两个普通空格。
- 不改变 Shift-Tab 反缩进规则，仍删除一个 tab、两个行首空格或一个行首空格。
- 不改变 CR/CRLF、尾随空行、emoji / UTF-16 光标边界、Markdown 片段插入、回车续写、预览解析、统计口径、JSON 持久化或 Mac Catalyst 构建方式。
- 不引入第三方依赖，不新增独立 native macOS target。

## 架构依据

- `MarkdownLineIndentation` 是 Tab / Shift-Tab 行缩进的唯一纯规则模块。
- `MarkdownBodyTextView`、`EntryEditorView` 和 Mac Catalyst 写作入口只能消费 `MarkdownLineIndentation.Result`，不得复制缩进规则。
- `JournalStore` 仍是唯一集合修改和本地持久化入口；本轮不得触碰保存格式。

## 实现步骤

1. 在 `MarkdownLineIndentation` 中新增私有 helper，例如 `bodyByApplying(_:to:)`。
2. helper 使用升序 `operations`，从原始 `body.startIndex` 开始追加未变片段和 replacement，最后追加剩余尾部。
3. 预估并 `reserveCapacity`，但不能依赖 UTF-16 数量作为精确字符数语义。
4. `apply(...)` 改为调用新 helper，不再对同一个 `String` 做多次反向 `replaceSubrange`。
5. 保持 operation range 全部来自原始 `body`，不要在构造过程中用已变更正文的 index。
6. 新增测试覆盖长多行选区的混合两空格、tab、单空格和不可反缩进行，断言正文和 `selectedRange`。
