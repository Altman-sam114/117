# v0.22（Markdown 引用回车续写）

## 版本号

`v0.22`

## 版本分配依据

当前最新正式版本为 `v0.21`。本轮继续沿 `v0.x` 小步优化，聚焦 Markdown 正文写作效率，不改变数据模型、持久化或完整统计看板。

## 背景

`MarkdownBodyTextView` 已经把正文输入集中到 `UITextView` bridge，并用 `MarkdownLineContinuation` 处理列表和待办的回车续写。项目同时已经支持引用片段、引用预览和引用工具栏入口，但正文里手写 `> ` 引用时，按回车不会自动续写引用前缀，空引用行也不会像空列表项一样退出当前结构。Mac Catalyst 和硬件键盘写作时，这会让连续引用记录变慢。

## 目标

- 扩展 `MarkdownLineContinuation`，支持 Markdown 引用行回车续写。
- 非空引用行按回车时延续相同缩进和 `> ` 前缀。
- 空引用行按回车时删除当前引用前缀，退出引用。
- 光标在引用行中间时按回车拆分当前行，并让下一行继续引用。
- 保持已有列表、待办、IME marked text、Tab / Shift-Tab 和片段插入行为不变。
- 新增或扩展 `MarkdownLineContinuationTests`，覆盖引用续写、退出、缩进、行中拆分、fenced code、非折叠选区、普通输入和 UTF-16/emoji 光标边界。
- 将 CI artifact 版本更新为 `v0.22`，并同步 README、测试规范、核心流程、流程图和更新日志。

## 非目标

- 不支持裸 `>` 引用；本轮只支持与当前 parser 和工具栏契约一致的 `> `。
- 不新增有序列表续写，不扩展 Markdown 预览语法。
- 不改变 `MarkdownSnippet` 文案、快捷键或工具栏顺序。
- 不改变 `JournalStore` 保存、排序、节流或 JSON 持久化。
- 不引入第三方依赖，不新增 native macOS target。

## 架构依据

- `MarkdownBodyTextView` 只负责 UIKit bridge 和输入事件入口，不承载 Markdown 字符串规则。
- `MarkdownLineContinuation` 属于规则层，消费正文、UTF-16 `NSRange` 和 replacement text，输出新正文与新选区。
- 规则层行为必须用 XCTest 覆盖；SwiftUI / UIKit 入口只调用纯规则。
- Mac Catalyst 仍走现有 target 和 GitHub Actions Mac Catalyst build 验证。

## 实现步骤

1. 在 `MarkdownLineContinuation` 中把当前列表前缀抽象扩展为行延续前缀：
   - 保留现有列表和待办识别、续写、退出逻辑。
   - 增加 `> ` 引用识别，允许行首空格或 tab 缩进。
   - 引用续写使用同缩进的 `> `。
   - 空引用行退出时删除当前行的缩进和 `> ` 前缀。
2. 保持 `MarkdownBodyTextView.Coordinator.shouldChangeTextIn` 的入口不扩大：
   - marked text 存在时继续走系统默认行为。
   - 只有纯规则返回结果时才手动写回 text view、正文 binding 和 selectedRange binding。
3. 扩展 `MarkdownLineContinuationTests`：
   - 非空引用行末尾续写。
   - 空引用行退出。
   - 光标在引用行中间拆分。
   - 缩进引用保留缩进。
   - fenced code block 内不触发。
   - 非折叠选区和非回车输入不触发。
   - emoji / UTF-16 光标保持正确。
   - 已有列表和待办用例继续保留。
4. 更新 workflow `VERSION: v0.22`。
5. 同步 README、测试规范、核心流程、流程图和更新日志。

## 本地轻量检查要求

人工已明确要求不要在本机运行构建、测试或 app。因此 Agent B 不执行本机 `xcodebuild build`、`xcodebuild test`、模拟器或 app 启动；最终验收只以 GitHub Actions 回传 artifact 为准。

可记录只读命令、git 状态、diff 复核和提交/推送命令；不得用本地输出冒充云端验收。

## main 直推要求

- 默认分支：`main`。
- 目标远端：`origin/main`。
- 开始前执行 `git fetch origin`、`git pull --ff-only origin main`。
- 不创建候选分支，不创建 PR。

- workflow 名称：`MD Journal CI Results`。
- artifact 名称应包含 `v0.22`、`main`、短 SHA、run id 和 run attempt。
- 结果包至少包含 manifest、failure summary、主日志、JUnit 和 `.xcresult`。

## Agent C 验收要求

Agent C 必须下载最新 `origin/main` commit 对应的未加密 artifact，并核对：

- manifest 中 `version=v0.22`。
- `branch=main`。
- `commitSha` 等于最新 `origin/main`。
- `runId`、`runAttempt` 与实际 GitHub Actions run 一致。
- `staticChecksOutcome`、`buildOutcome`、`macCatalystBuildOutcome`、`testOutcome` 均为 `success`。
- `junit.xml` failures 为 0，skipped 为 0。
- `xcodebuild.log` 和 `maccatalyst-build.log` 包含 `** BUILD SUCCEEDED **`。
- `xctest.log` 包含 `** TEST SUCCEEDED **`，且能看到 `MarkdownLineContinuationTests` 执行。
- 三个 `.xcresult/Info.plist` 均可解析。
